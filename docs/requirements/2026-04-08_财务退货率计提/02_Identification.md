# 财务退货率 / 退款计提 — 原始需求识别（02_Identification）

> 输入基线：`01_Parsing.md`（2026-04-08）。本文档输出：用户故事、功能清单、业务规则、边界与异常。  
> 角色：跨境电商 ERP / 业财一体化视角。

---

## 1. 用户故事 (User Stories)

### US-1 全店铺自动化测算（替代人工导数手算）

**作为** 财务人员，**我想要** 系统按月自动汇总约 40 个店铺的退款与收入口径数据并计算退款率，**以便** 不再依赖人工向建宏要数、手算，且满足计提对精度的要求。

- **AC**
  - 支持按店铺、按月触发或定时跑批（至少月维度）。
  - 跑批结果可复现（同输入、同规则下结果一致）。
  - 覆盖「此前仅部分大渠道/亚马逊美国」扩展到业务约定的全店铺范围。

### US-2 亚马逊：报告驱动归属与退款金额

**作为** 财务人员，**我想要** 系统基于亚马逊 Transaction（`type=refund`）与历史 Order 报告自动匹配「确认收入月」并汇总退款金额，**以便** 与平台结算口径一致、支撑订单月退款率。

- **AC**
  - 退款行金额 = 销售退货（销售额 + 运费收入 + 包装收入）+ 退货（折扣），符号与报告原始字段一致。
  - 亚马逊侧不纳入平台税额（与业务约定一致）。
  - 每条退款可关联到订单及 **MSKU** 粒度（若报告支持行级；若仅订单级需有明确分摊规则见 BR）。

### US-3 非亚马逊（Shopify / App 商城等）：金蝶确认收入 + 收单退款

**作为** 财务人员，**我想要** 非亚马逊渠道的「确认收入日期」以金蝶为准、退款来自已对接收单数据，并可用金蝶含税价与税额推算不含税参与计算，**以便** 业财口径统一且满足自营不含税测算要求。

- **AC**
  - 收入侧日期字段来源标注为金蝶（与亚马逊报告源区分展示）。
  - 收单退款可从现有对接拉取；不含税可按金蝶含税价、税额推算（公式见 BR）。
  - 渠道维度不与亚马逊混用两套日期逻辑。

### US-4 多币种与换汇展示

**作为** 财务人员，**我想要** 在店铺维度查看原币金额，并同时展示按规则换算后的 USD 与 CNY，**以便** 分店铺核算与合并看数。

- **AC**
  - 收入相关金额使用 **确认收入月** 月度汇率；退款相关金额使用 **退货发生月** 月度汇率。
  - 报表与明细中同时保留原币与折算币别字段（或列）。

### US-5 订单月退款率 + 趋势与 MSKU 下钻

**作为** 财务或运营分析人员，**我想要** 按 **订单月** 查看退款率，并按店铺、年/月趋势查看，且能下钻到 **MSKU**，**以便** 做店铺级计提与商品级分析。

- **AC**
  - 指标定义在系统中可配置或文档化（分子/分母与 01 中待确认项对齐前，需占位或默认方案）。
  - 支持按店铺、时间维度筛选与图表/表格趋势（实现形态可为报表页 + 导出）。

### US-6 历史补算（创建月滚动 + 发货下界）

**作为** 财务人员，**我想要** 对历史订单按 **订单创建月** 滚动补算，且仅处理 **发货 ≥ 2025-01** 的数据，**以便** 2025→2026 计提与回溯一致。

- **AC**
  - 支持按创建月批次重跑或初始全量补算（与性能/幂等设计配合）。
  - 边界订单（创建月与发货月下界冲突）在规则确认后系统行为可配置或按财务签字规则固定。

### US-7 报表 + 明细下载（不对接凭证）

**作为** 财务人员，**我想要** 在线查看汇总报表并下载明细行，**以便** 留档、与金蝶或 Excel 二次核对；本期不要求推送凭证。

- **AC**
  - 明细包含关键审计字段：渠道、店铺、MSKU、订单标识、订单月、确认收入月、退货发生月、原币金额、USD、CNY、数据来源（报告/金蝶/收单）等。
  - 导出格式至少一种通用格式（如 CSV/Excel）。

## 3. 业务规则 (Business Rules)

### BR-1 渠道数据源（权威单一）

- **亚马逊**：订单、收入归属、退款明细以**平台报告**为唯一依据；日期以报告字段为准。
- **Shopify、App 商城等**：**确认收入时间**以**金蝶**为准；不得与平台前台日期混用作收入确认。
- **自营退款金额**：以**已对接第三方收单**数据为准；不含税参与计算时结合 **BR-3**。

### BR-2 亚马逊退款金额

- 筛选条件：`transaction` 报告中 `type = refund`（或业务等价字段）。
- **单行退款计入额** = 销售退货（销售额 + 运费收入 + 包装收入）+ 退货（折扣）；各项取自报告字段。
- **符号**：与报告**原始金额符号**一致，不在系统内擅自取绝对值改变方向。
- **税**：亚马逊报告税额已拆分，本需求**不计入**上述组合（除非财务书面变更口径）。

### BR-3 亚马逊「确认收入月」

- 对每笔退款关联订单，在**历史 order 类型**报告中取对应记录的**月报告**与 **date/time**，推导该订单的**确认收入月份**（与财务定义一致前提下实现）。

### BR-4 非亚马逊不含税推算

- 在具备金蝶 **含税价**、**税额** 的前提下：  
  - 税率推算需与财务确认（例如：税额/（含税价−税额）或价内税公式等），**以金蝶科目与价税方式为准**。
- 参与退款率/计提的金额为**不含税金额**时，分子分母需**同为不含税或同为可比口径**（禁止分子不含税、分母含税未说明）。

### BR-5 换汇

- **收入类金额**（分母及与收入同月认定的销售额等）：使用 **确认收入月** 对应**月度记账汇率**。
- **退款类金额**（分子）：使用 **退货发生月** 对应**月度记账汇率**。
- 币种路径：店铺原币 → USD、CNY；中间价或公司规定牌价以财务主数据为准。

### BR-6 历史补算范围

- **滚动维度**：按 **订单创建月** 分批补算或重算。
- **下界**：仅包含 **订单发货日期 ≥ 2025-01-01**（或业务约定的当月首日）的订单；与创建月的交集规则以财务最终签字为准（见 Edge Cases）。

### BR-7 退款率（订单月）

- **统计维度**：以 **订单月** 为展示与核心计算维度之一（与 01 一致）。
- **分子/分母**：具体是否含运费、包装、折扣与 MSKU 分摊，**依赖 01 中待确认项**；上线前需固化为一版「计提版退款率定义」并写入配置或字典表。

### BR-8 交付边界

- 本期输出为 **报表 + 明细下载**；**不**自动生成金蝶凭证、不推送总账。

## 4. 潜在异常与边界 (Edge Cases)

### 4.1 数据缺失与匹配失败

- 亚马逊 **refund** 无法在 Order 报告中找到对应订单或 date/time 缺失 → 需标记异常行，不计入汇总或计入「待人工核对」队列，避免静默丢数。
- 金蝶侧无确认收入日期、含税价/税额为空 → 该行不参与不含税推算或整单拦截，需告警。
- 收单有退款但金蝶无对应冲减（或反之）→ 对账差异列表。

### 4.2 口径与边界冲突

- **创建月在 2024、发货在 2025-01**：是否纳入补算、归属哪一创建月批次 → 未决前系统须有默认策略（如排除/纳入并打标）避免重复结论。
- **创建在 2025、发货早于 2025-01**：是否排除需规则固化。
- **退货发生月**（非亚马逊）：取收单时间还是金蝶凭证日期未决 → 换汇结果可能不同，需双字段预留或规则开关。

### 4.3 多币种与汇率

- 某月缺失汇率 → 禁止静默用上月；应失败告警或使用财务指定的备用规则（需书面）。
- 原币小数位与四舍五入：汇总层与明细层舍入顺序需统一，防止「明细加总≠报表一行」。

### 4.4 亚马逊报告特性

- 报告延迟、重拉修正 → 跑批需支持**按区间重算**与版本留痕，避免与上月已导出明细永久不一致且无解释。
- 部分费用项字段名变更或区域差异 → 解析层需可配置映射。

### 4.5 MSKU 粒度

- 退款仅在订单级、无行级 MSKU → 需明确分摊规则（按发货行数量/金额占比/主 MSKU），否则 MSKU 报表会产生系统性偏差。

### 4.6 性能与作业

- 40 店铺 × 多报告 × 历史从 2025 起 → 全量补算可能超时；需分批、断点续跑、限流。
- 并发跑批与人工重跑同一月份 → 需幂等键（店铺+渠道+订单月+批次号）防止重复入账式汇总。

### 4.7 外部依赖

- 金蝶/收单 API 超时、部分成功 → 重试策略与「部分失败」展示；不得产出半套无标识的「完整」报表。

### 4.8 合规与审计

- 导出明细含个人隐私或平台敏感字段时，需权限控制与下载水印/日志（按公司规范）。
